DDD 核心概念与战略设计

这是 DDD 系列文章的第一篇，主要介绍 DDD 的核心概念与战略设计。后续文章将深入战术设计和 SpringBoot3 实战。

一、什么是 DDD？为什么需要 DDD？

1.1 传统开发的问题

在传统的软件开发中，我们经常遇到以下问题：

  • 业务逻辑分散：业务规则散落在 Controller、Service、DAO 各层
  • 技术代码污染：领域代码中充斥着数据库、框架相关的技术细节
  • 沟通鸿沟：开发人员与业务专家使用不同的语言，导致需求理解偏差
  • 代码难以维护：随着业务发展，代码越来越复杂，修改一处可能影响多处

1.2 DDD 的核心思想

领域驱动设计（Domain-Driven Design，简称 DDD） 是由 Eric Evans 在 2003 年提出的一种软件开发方法论。其核心思想是：

将业务领域作为软件设计的核心，让代码结构反映业务结构。

DDD 强调：

  • 开发人员与业务专家紧密协作
  • 建立统一的通用语言（Ubiquitous Language）
  • 专注于领域模型的设计与实现
  • 通过限界上下文（Bounded Context）控制复杂度

1.3 何时使用 DDD？

DDD 并非银弹，适用于以下场景：

✅ 适合使用 ❌ 不适合使用
业务逻辑复杂 简单的 CRUD 应用
长期演进的项目 一次性或短期项目
需要多人协作 个人小项目
业务规则频繁变化 需求稳定的项目

二、核心概念

2.1 领域（Domain）

领域 是指业务所涉及的知识和规则的总和。例如：

  • 电商领域：商品、订单、支付、物流
  • 金融领域：账户、交易、风控、结算
  • 博客领域：文章、分类、标签、评论

2.2 子域（Subdomain）

复杂的领域可以拆分为多个子域。DDD 将子域分为三类：

（1）核心域（Core Domain）

  • 企业的核心竞争力所在
  • 需要投入最多资源
  • 例如：淘宝的交易核心、抖音的推荐算法

（2）支撑域（Supporting Domain）

  • 支持核心域运作
  • 可以内部开发
  • 例如：用户管理、权限系统

（3）通用域（Generic Domain）

  • 通用功能，非差异化竞争点
  • 建议直接购买或使用开源方案
  • 例如：邮件发送、短信通知、文件存储

2.3 限界上下文（Bounded Context）

限界上下文 是 DDD 中最重要的概念之一，它定义了模型的边界和适用范围。

为什么需要限界上下文？

同一个概念在不同场景下可能有不同含义：

限界上下文的设计原则

  1. 单一职责：每个上下文有明确的业务边界
  2. 高内聚低耦合：上下文内部高度凝聚，上下文之间松耦合
  3. 团队对齐：一个上下文最好由一个团队负责

三、通用语言（Ubiquitous Language）

3.1 什么是通用语言？

通用语言 是开发人员与业务专家共同使用的、精确的、无歧义的语言。

3.2 通用语言的特点

  • 精确性：每个术语有明确的定义
  • 一致性：代码命名与业务术语一致
  • 演进性：随着业务理解深入而演进

3.3 实战示例

3.4 建立通用语言的步骤

  1. 收集术语：与业务专家访谈，记录关键术语
  2. 定义术语：为每个术语编写明确的定义
  3. 统一使用：在代码、文档、沟通中一致使用
  4. 持续演进：定期 review 和更新术语表

五、上下文映射模式

当多个限界上下文需要协作时，DDD 定义了多种映射模式：

5.1 合作关系（Partnership）

两个上下文相互依赖，共同演进。

5.2 共享内核（Shared Kernel）

两个上下文共享部分领域模型。

5.3 客户 - 供应商（Customer-Supplier）

下游（客户）依赖上游（供应商），上游优先考虑下游需求。

5.4 防腐层（Anti-Corruption Layer）

通过适配层隔离外部系统，防止外部模型污染内部模型。

5.5 开放主机服务（Open Host Service）

将功能作为服务暴露给其他上下文使用。

5.6 遵从者（Conformist）

下游直接采用上游的模型，不做转换。

七、总结

7.1 核心要点回顾

概念 作用 关键点
子域 分解复杂领域 核心域优先投入
限界上下文 控制模型边界 一个上下文一个模型
通用语言 统一沟通 代码与业务术语一致
上下文映射 上下文协作 根据实际情况选择模式

7.2 下一步

战略设计完成后，我们需要深入战术设计，学习如何用代码实现领域模型：

  • 实体（Entity）：有唯一标识的对象
  • 值对象（Value Object）：通过属性定义的对象
  • 聚合根（Aggregate Root）：一致性边界
  • 仓库（Repository）：领域对象的持久化抽象
  • 领域服务（Domain Service）：跨实体的业务逻辑
