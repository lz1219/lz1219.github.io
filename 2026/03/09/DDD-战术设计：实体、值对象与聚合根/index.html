DDD 战术设计：实体、值对象与聚合根

这是 DDD 系列文章的第二篇，主要介绍 DDD 的战术设计核心概念。上一篇介绍了 战略设计，下一篇将进入 SpringBoot3 实战。

一、战术设计概述

1.1 战略设计 vs 战术设计

1.2 战术设计的目标

  • 实现领域模型：将战略设计的概念转化为代码
  • 保持模型纯净：业务逻辑与技术细节分离
  • 保证一致性：通过聚合根维护业务规则
  • 提高可测试性：领域代码独立于框架

二、实体（Entity）

2.1 什么是实体？

实体 是具有唯一标识和连续性的领域对象。实体的核心特征是标识性，即使属性相同，两个实体也是不同的。

2.2 实体的特征

特征 说明
唯一标识 每个实体有唯一的 ID
连续性 生命周期内标识不变
可变性 属性可以变化
行为 包含业务逻辑

2.3 实体设计示例

2.4 实体 ID 设计

三、值对象（Value Object）

3.1 什么是值对象？

值对象 是通过属性来定义的对象，没有唯一标识。值对象的核心特征是概念完整性和不可变性。

3.2 值对象的特征

特征 说明
无标识性 通过属性值判断相等
不可变性 创建后不能修改
自验证 构造时验证有效性
可替换性 可以整体替换

3.3 值对象设计示例

3.4 常见的值对象类型

3.5 实体 vs 值对象

对比项 实体 值对象
标识 有唯一 ID 无 ID，通过属性判断相等
可变性 通常可变 不可变
生命周期 有明确生命周期 临时存在
设计重点 标识一致性 概念完整性
示例 用户、订单 地址、金额、邮箱

四、聚合根（Aggregate Root）

4.1 什么是聚合根？

聚合 是一组相关对象的集合，这些对象作为一个整体被对待。聚合根 是聚合的入口点，外部对象只能通过聚合根访问聚合内的其他对象。

4.2 聚合的设计原则

4.3 聚合根的核心原则

原则 说明
不变性边界 聚合内保证数据一致性
唯一入口 外部只能通过聚合根访问
内部引用 聚合内对象可以有内部引用
外部引用 外部对象只能引用聚合根 ID
事务边界 一个聚合是一个事务边界

4.4 聚合根设计示例

4.5 聚合设计常见错误

4.6 聚合大小的判断标准

指标 合理范围
实体数量 3-5 个
方法数量 10-20 个
代码行数 300-500 行
事务时间 < 1 秒

五、仓库（Repository）

5.1 什么是仓库？

仓库 是领域对象的持久化抽象，它提供了类似集合的接口来访问领域对象，屏蔽了底层持久化细节。

5.2 仓库的设计原则

5.3 仓库接口设计

5.4 仓库实现（基础设施层）

5.5 MyBatis-Plus 集成

六、领域服务（Domain Service）

6.1 什么是领域服务？

领域服务 用于封装不属于任何实体的业务逻辑。当一段业务逻辑涉及多个聚合或无法自然归属到某个实体时，使用领域服务。

6.2 领域服务的使用场景

场景 说明
跨聚合操作 涉及多个聚合根的协作
外部依赖 需要调用外部服务
复杂计算 复杂的业务计算逻辑
临时操作 不属于任何实体的临时操作

6.3 领域服务设计示例

6.4 领域服务 vs 应用服务

对比项 领域服务 应用服务
位置 领域层 应用层
职责 核心业务逻辑 协调、编排
依赖 只依赖领域对象 可依赖领域服务、外部服务
事务 不包含事务 通常包含事务边界
示例 订单创建逻辑 订单创建流程编排

七、总结

7.1 战术设计要点回顾

7.2 设计原则

  1. 充血模型：实体包含业务逻辑，不是贫血模型
  2. 不可变值对象：值对象创建后不能修改
  3. 小聚合：保持聚合小巧，一个事务边界内完成
  4. 依赖倒置：领域层定义接口，基础设施层实现
  5. 通用语言：代码命名与业务术语一致
